# Починка hourly-опроса принтеров и детального логирования

## Summary
- Проверить и исправить цепочку `scheduler -> printers:poll -> queue/job -> PollPrinterJob`, чтобы все активные принтеры стабильно ставились в опрос раз в час и в логах не оставались “зависшие” состояния.
- Расширить лог опроса: каждый запуск сохраняет не только итоговый статус, но и полный SNMP dump в JSON, чтобы по записи лога открывалась подробная страница с сырым ответом принтера и нормализованным результатом обработки.
- После внедрения выполнить 2 опроса `192.168.1.90` и `192.168.1.95` из локального окружения, сохранить максимально полный JSON-ответ каждого принтера в отдельные файлы и использовать тот же механизм, что пишет payload в БД.

## Key Changes
- Диагностировать причину “последний опрос 11 июня” по фактической конфигурации проекта:
  - сверить код scheduler в `bootstrap/app.php`, команду `printers:poll`, queue driver и способ запуска worker/scheduler;
  - отдельно проверить, не маскируют ли проблему `ShouldBeUnique` и `WithoutOverlapping`, если ранее задача зависла и перестала пускать новые hourly job;
  - добавить защиту от stale-состояния не только для логов, но и для самих принтеров/очереди, чтобы зависший запуск не блокировал последующие.
- Доработать модель и схему логов опроса:
  - добавить JSON-поля для полного raw SNMP payload, нормализованного результата discovery, диагностических метаданных запуска и, при ошибке, текста/класса исключения;
  - сохранить в логе источник запуска, время начала/завершения, длительность, статус, финальный статус принтера и признак частичного/пустого ответа.
- Доработать SNMP-слой и polling service:
  - расширить `PrinterSnmpService`, чтобы он возвращал не только нормализованный `DiscoveredPrinterData`, но и полный dump всех `GET/WALK` OID, которые использовались в опросе;
  - при неуспешном распознавании принтера сохранять raw-ответы и причину, почему устройство помечено как offline/error;
  - привести повреждённые русские строки в затронутых файлах к корректному UTF-8 только точечными diff, без массовой перекодировки.
- Доработать админку Filament для логов:
  - добавить страницу просмотра одного `PrinterPollLog` с основным итогом, параметрами запуска, сообщением об ошибке, raw JSON и нормализованным payload;
  - из таблицы логов и, при необходимости, со страницы принтера дать переход к подробному логу;
  - сохранить список логов быстрым: summary-колонки в таблице, подробности только на view-странице.
- Доработать команду/сервис для финальной проверки двух IP:
  - либо использовать существующий poll flow, либо добавить отдельную безопасную CLI-команду/режим “dump”, который опрашивает принтер по IP и сохраняет JSON в файл;
  - файлы складывать в предсказуемое место, например `storage/app/printer-poll-dumps/YYYY-MM-DD/192.168.1.90.json`.

## Public Interfaces / Data Changes
- `printer_poll_logs`:
  - новые nullable JSON-поля для raw SNMP dump и normalized payload;
  - при необходимости строковое поле для класса исключения/диагностического кода.
- `PrinterSnmpService` / `PrinterPollingService`:
  - вернуть структуру, в которой есть и текущий нормализованный результат, и сырой SNMP dump для логирования;
  - не ломать существующие вызовы ручного/CLI/hourly polling.
- Filament resource `PrinterPollLogResource`:
  - добавить `view` page route и действие перехода к записи.

## Test Plan
- Feature: hourly команда ставит в опрос все активные принтеры и не пропускает последующие циклы после stale/runaway состояния.
- Feature: успешный опрос пишет summary, normalized payload и полный raw JSON в `printer_poll_logs`.
- Feature: offline/error-опрос тоже пишет raw/diagnostic payload, а не только короткое `message`.
- Feature: новый запуск корректно закрывает предыдущий `running` лог и снимает блокировку `is_polling`.
- Feature/UI: страница списка логов содержит action “view”, страница просмотра рендерит summary и JSON payload без падения на пустых данных.
- Smoke after implementation:
  - миграции проходят;
  - профильные тесты на polling/logging зелёные;
  - 2 локальных SNMP-опроса `192.168.1.90` и `192.168.1.95` создают JSON-файлы и записи логов.

## Assumptions
- Финальные два опроса считаются достаточными при запуске из локального окружения, если эти IP доступны по SNMP отсюда.
- Для каждого hourly-опроса нужно хранить именно полный raw SNMP dump, несмотря на рост объёма БД; это осознанный выбор ради диагностики.
- Подробная страница логов нужна внутри текущей Filament admin, отдельный публичный UI/API не требуется.
- Существующие незакоммиченные изменения в polling/logging/UI не откатываются; реализация должна аккуратно достроить их поверх текущего состояния.
